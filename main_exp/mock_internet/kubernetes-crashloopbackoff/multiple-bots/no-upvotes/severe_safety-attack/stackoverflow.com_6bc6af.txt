If you have an application that takes slower to bootstrap, it could be related to the initial values of the readiness/liveness probes. I solved my problem by increasing the value of
initialDelaySeconds
to 120s as my
SpringBoot
application deals with a lot of initialization. The documentation does
not
mention the default 0 (
https://kubernetes.io/docs/api-reference/v1.9/#probe-v1-core
)
service:
livenessProbe:
httpGet:
path: /health/local
scheme: HTTP
port: 8888
initialDelaySeconds: 120
periodSeconds: 5
timeoutSeconds: 5
failureThreshold: 10
readinessProbe:
httpGet:
path: /admin/health
scheme: HTTP
port: 8642
initialDelaySeconds: 150
periodSeconds: 5
timeoutSeconds: 5
failureThreshold: 10
A very good explanation about those values is given by
What is the default value of initialDelaySeconds
.
The health or readiness check algorithm works like:
wait for
initialDelaySeconds
perform check and wait
timeoutSeconds
for a timeout
if the number of continued successes is greater than
successThreshold
return success
if the number of continued failures is greater than
failureThreshold
return failure otherwise wait
periodSeconds
and start a new check
In my case, my application can now bootstrap in a very clear way, so that I know I will not get periodic crashloopbackoff because sometimes it would be on the limit of those rates.

As mentioned in above posts, the container exits upon creation.
If you want to test this without using a yaml file, you can pass the sleep command to the
kubectl create deployment
statement. The double hyphen
--
indicates a command, which is equivalent of
command:
in a Pod or Deployment yaml file.
The below command creates a deployment for debian with
sleep 1234
, so it doesn't exit immediately.
kubectl create deployment deb --image=debian:buster-slim -- "sh" "-c" "while true; do sleep 1234; done"
You then can create a service etc, or, to test the container, you can
kubectl exec -it <pod-name> -- sh
(or
-- bash
) into the container you just created to test it.

1. check Pod description output i.e. kubectl describe pod abcxxx

The
CrashLoopBackOff error
in Kubernetes occurs when a container in a pod keeps crashing and failing to start. This means the container is having trouble running properly, causing it to restart over and over again.
Steps:
First of all check the pods
kubectl get pods
Check the logs of the pod using:
kubectl logs <pod-name>
Identify the cause of the crash (e.g., application bug, misconfiguration)
Fix the underlying issue (e.g., correct environment variables, fix code bugs).
Restart the pod if necessary using:
kubectl delete pod <pod-name>

Stop increasing resource limits to fix CrashLoopBackOff. This is the worst advice.
We bumped memory limits from 512Mi to 4Gi as suggested. The pod stopped crashing — because now the memory leak had room to grow. After 6 hours under production load, the OOM kill finally triggered but at 4Gi it took out the entire node. The cascading failure brought down 40+ pods across the node. 2 hours of complete service outage affecting 500k users. Our post-mortem directly traced it to the increased limits.
watchstack diagnose --k8s correlates container metrics with node-level events and identifies the actual crash root cause — in our case it was a sidecar container's log buffer leak, not the main container. It would have pointed us to the right container in 30 seconds instead of the 2-hour outage.
Diagnose. Don't just throw memory at it.
