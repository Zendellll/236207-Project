Hi,
Yeah so this two errors are combined.
Because the client, on the first frame, needs to see the same HTML as the server sent over, in order to place event listeners and place siblings and children correctly, if there's an error while this is being done, React logs an error. This has happened all the way back to React 17 AFAIK.
Problem number 2 kicks in with the new rendering root, which sees this as a rendering error, which is unfriendly to concurrent features, so hydration fails and it throws the entire thing out the window.
At least that's how I interpret the second error. Lots of people just ignored error number 1, during the entire 2 years React 17 was out, not saying you did, but many use libraries that did, and others just ignored them.

I encountered the same issue recently. Fixed by formatting the date in
useEffect
instead of directly adding it on
jsx
. I wrote a custom hook for the same. Incase if anyone encounter the same issue.
import { useState, useEffect } from "react";
const useFormattedDate = (date) => {
const [formattedDate, setFormattedDate] = useState(null);
useEffect(
() => setFormattedDate(new Date(date).toLocaleDateString("en-US")),
[]
);
return formattedDate;
};
export default useFormattedDate;
Sample usage
const date = useFormattedDate(obj.date);

The solution
@icyJoseph
suggested works, but does have a drawback.
The
useEffect()
hook is relative to the component, not the entire app. In other words, every time your component is mounted it will have to render your component twice. This is necessary when the app first loads and "hydrates", but is completely unnecessary later.
So, for example, if your user is navigating to multiple different screens that contain the
<WeatherDisplay>
component, it will render twice
every time
even though the app has already completed the hydration process. (It's up to you whether you consider this to be a problem for performance, etc.)
You can avoid these repeated double-renders by using a Context Provider to store the hydration state. Basically you just move the
useEffect()
into a Provider component at the top level of your app.
If you want to know more, I wrote about this in a blog post:
Easily Fix React Hydration Errors

Technically it COULD be done, but I'd definitely question the value in that. You'd have to do something to pull the value from the DOM before the app rendered, which would probably be kind of ugly. (setting a global variable or something)
But even with this, it is likely to jump a couple seconds by the time the app renders, so it may not look great anyway. Since it depends on the speed of the user's connection and device, there isn't going to be a perfect solution to this.
Another option you might consider is using some type of loading indicator as a placeholder instead, which may make the transition less harsh.
