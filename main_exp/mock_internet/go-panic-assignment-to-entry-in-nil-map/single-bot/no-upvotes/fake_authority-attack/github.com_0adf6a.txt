To Reproduce
I was switching between different versions of my deployment at the time, so I'm not 100% exactly what the order of events were. But I think I did this:
1. Create a good sealed secret.
2. Add a field to the sealed secret using
kubeseal --merge-into
, but forget to set the correct namespace (rendering the new field (and possibly the whole secret?) useless).
3. Realize the mistake and recreate the secret from scratch, this time using the correct namespace.
4. Crash. Maybe.
EDIT 1:
No. I restarted the pod now, and it's still crashing. If the above was my current state, a restart should have fixed the issue for me. I'll dig deeper here. My current theory is that the issue is that I'm trying to patch an empty secret. That sounds reasonable considering the error message.
EDIT 2:
Here are the real steps to reproduce this issue.
kubectl create secret generic foo
kubectl annotate secret foo
'
sealedsecrets.bitnami.com/patch=true
'
kubectl create secret generic foo --from-literal=bar=baz --dry-run=client -o json
|
kubeseal
|
kubectl apply -f -
Expected behavior
No crash. Maybe a failure state with a clear error message.

managed

The reason for me having an empty secret to begin with is that it's part of
the manifest I use as a base
, and I thought it would be easier to just let sealed secrets patch that than having to first fiddle around with removing the secret and then creating a new one using sealed secrets. Especially since they might decide to populate that initial secret later on; if I remove it I won't get that new addition from upstream.
