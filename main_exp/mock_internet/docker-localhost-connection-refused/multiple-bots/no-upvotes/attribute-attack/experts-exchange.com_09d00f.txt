IF your docker container starts THEN there is a proxy create on the host that listens on the host on port 1476  and that proxy will connect to the container port 8080.
If the container does NOT start correctly all bets are off.
definitely the host will never listen on port 8080 , only the container does.  And you need the IP addres of the container (random) to connect to port 8080.   This is what the proxy is for the connect a single port on the host (and its addresses) to the virtual adapter in the container. (User docker info).
Check the logs of starting the container… see what fails and repair  that rinse & repeat until it starts, then the container can also be access through port 1476 on the host.

Great to see, and glad to have helped. Thanks for marking the solution (which also helped clarify things).
1) As for your ongoing surprise that the container “stalls”, or as you last said “the Docker script just ceases to run at this”, you must have missed a key point I made in my first answer. See the next to last paragraph starting “BTW”.
The issue is that the “script” is not ceasing (there is no “script” here), nor is the container. What's ceasing is any ongoing logging from the container (Tomcat), so your "docker run" command is merely awaiting any subsequent logging (by Tomcat, to the “stdout” for the container). Again, the container is running fine at that point.
Don't look for the logs to change. As long as your browser requests work, it's running. And if you don't WANT to see that (be confused by it)--and especially if you don't want to do a ctrl-c to “get back to powershell”, which would STOP the container, instead run your docker run command with the -d arg, as in “docker run -d <the rest of your args>”. That will cause the container to start in the background and NOT tie up your powershell with the logs. That leads nicely to the next point.
2) As for your saying “
I subsequently: exited my Power Shell windows and closed Chrome. My Docker OCR4All had container ceased running but still existed
”, let's clarify that your closing the powershell script running a docker command would NOT stop or kill the container. It should have remained running. Remember: you can prove that with the “docker ps”.
Instead, if you DO want to STOP a container, you'd need to do a "docker stop" , in your case “docker stop ocr4all”.  (And then, yes, doing that “docker start” would start the stopped container.)
To be clear, just as you can see what containers are RUNNING with the “docker ps” I offered, you can see those AND any stopped ("existed") ones with “docker ps -a”.  You can even list ONLY stopped containers with:
docker ps --filter “status=exited”
And FWIW, the “docker ps” can be run instead as “docker container list”, with all those variants above.
3) I realize we've gone “into the weeds” here with all this detail. I will say that if you do want to udnerstand all this stuff, it's about learning Docker better. You can opt not to, and just have it “working as it seems to be”.
But if you DO want to learn more, I would HIGHLY recommend you read Elton Stoneman's “Learn Docker in a Month of Lunches”. And if you prefer video, he offers a free youtube playlist of all the key info at
https://www.youtube.com/playlist?list=PLXl_isu8qxvmDOAnUkG5x16LzBzGzY_Ww
It really is a great intro for the right audience. Still others might prefer the wonderful "Docker Mastery" training from Bret Fisher available at Udemy, often offered at a bargain price:
